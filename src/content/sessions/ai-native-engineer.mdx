## O que define um AI-Native Engineer

Se você tira a IA do seu dia e o trabalho continua igual, você está usando IA como acessório. Um AI-Native Engineer é alguém que reorganizou o fluxo inteiro — delegar, revisar, iterar — ao redor do que a IA faz bem. Isso significa três coisas:

1. **Saber o que delegar e o que não delegar**. IA é boa em gerar rascunhos, explorar opções, automatizar o repetitivo. IA é ruim em julgar trade-offs de negócio, entender contexto político de um time, ou decidir o que não construir.

2. **Ter processo, não só ferramenta**. A diferença entre um dev que usa Copilot e um AI-Native Engineer é a mesma diferença entre alguém que tem um martelo e alguém que sabe construir uma casa. Ferramenta sem processo é só barulho.

3. **Revisar com olho crítico**. IA gera código que compila, texto que parece correto, sugestões que fazem sentido na superfície. O trabalho do AI-Native Engineer é olhar por baixo da superfície.

Isso vale pra todo mundo: dev, QA, PM, designer. A diferença de maturidade não é técnica. É de mentalidade.

### O que mudou

Pensar ficou mais importante que digitar. Essa é a mudança central.

Antes de IA no fluxo de trabalho, um dev sênior se diferenciava por velocidade de digitação, memória de APIs, e pattern matching acumulado. Agora, velocidade de digitação é irrelevante (o modelo digita por você), memória de APIs é menos crítica (o modelo sabe a documentação), e pattern matching pode ser parcialmente delegado.

O que subiu de valor: capacidade de especificar com clareza, avaliar output criticamente, e entender o sistema como um todo.

### O que não mudou

Fundamentos continuam fundamentais. Quem não entende como um banco de dados funciona não vai saber avaliar se o código que a IA gerou vai travar em produção. Quem não entende requisitos não vai saber escrever uma spec que produza resultado útil.

IA amplifica o que você já sabe. Se você sabe pouco, ela amplifica a confusão. Se você sabe muito, ela amplifica a produtividade.

## Por que isso importa

A maioria dos profissionais de tecnologia já usa alguma ferramenta de IA no dia a dia. Autocomplete no editor, chat pra tirar dúvidas, geração de código a partir de descrições em texto. Mas usar uma ferramenta e operar com ela de forma nativa são coisas diferentes.

Um AI-Native Engineer não é alguém que sabe mais prompts ou que usa mais ferramentas. É alguém que reorganizou o fluxo de trabalho pra tirar proveito real do que a IA faz bem, e sabe identificar onde ela atrapalha.

Isso importa porque a diferença entre "usar IA" e "trabalhar de forma AI-native" é a diferença entre copiar respostas do ChatGPT e construir um sistema que funciona com agentes coordenados.

## Exemplo aplicado

Imagine que você precisa criar uma API de autenticação. Um dev que "usa IA" abre o chat, pede "cria uma API de auth com JWT", copia o resultado e cola no projeto. Talvez funcione. Talvez não.

Um AI-Native Engineer faz diferente:

1. Escreve uma especificação clara do que precisa (endpoints, fluxos, restrições)
2. Passa a spec pro agente de código com contexto do projeto
3. Revisa o output com olho crítico, não aceita tudo automaticamente
4. Usa a IA pra gerar testes a partir da spec original
5. Itera com feedback específico quando algo não bate

A diferença não é a ferramenta. É o processo.

## Onde isso quebra

Cuidado com esses anti-padrões:

- **Aceitar tudo sem revisar**: IA gera código plausível, não código correto. Se você aceita sem ler, está terceirizando a responsabilidade, não o trabalho.
- **Prompt como substituto de pensamento**: "faz pra mim" não é especificação. Quanto mais vago o pedido, mais genérico o resultado.
- **Achar que IA substitui experiência**: IA acelera quem já sabe o que quer. Pra quem não sabe, ela amplifica a confusão.

> Ferramenta boa com processo ruim produz resultado ruim mais rápido.

## O que levar daqui

- Na próxima vez que abrir uma ferramenta de IA, pare antes de digitar o prompt. Escreva primeiro o que você quer, com que restrições, e como vai avaliar o resultado.
- Comece por identificar onde IA realmente ajuda no seu trabalho — não onde é moda usar. Nem tudo precisa de IA, e saber isso já é maturidade.
- Revise todo output de IA como se um júnior tivesse escrito. Se você não consegue avaliar a qualidade, o sinal é claro: estude o fundamento, não a ferramenta.
