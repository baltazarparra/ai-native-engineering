## The shift that matters

The core problem in AI-assisted software work is not "which model should we use?" That changes constantly. The real problem is: **how do we turn human intent into reliable software without losing control, quality, and responsibility along the way?**

Vibe coding showed that models can generate a lot of code from natural language. That is powerful for prototypes. But in an enterprise environment, delivery cost is not just writing code. It is aligning scope, preserving architecture, protecting data, validating behavior, reviewing changes, auditing decisions, and keeping the system healthy after deployment.

Two ideas need to work together:

- **SDD**, or Spec-Driven Development: writing and evolving specifications that act as an intent contract.
- **Harness Design**: designing the work system around the agent so it has context, tools, boundaries, validation, and governance.

Separately, these ideas help. Together, they change the level of the workflow.

## Why this matters

A small team can tolerate more improvisation. A prototype can start from a short conversation with the model. A larger organization needs coherence. Different people need to request, execute, review, and maintain software with the same standard.

Without SDD, the agent works from ambiguous intent. It may generate something polished, functional, and wrong.

Without a harness, the spec becomes a document someone has to police manually. The agent may receive good instructions, but the system does not help prove that it followed them.

Professional AI-native development starts when the team stops thinking only about "better prompts" and starts designing a **delivery system**.

## What SDD is

SDD means **Spec-Driven Development**.

In this context, a spec is not a huge PDF and it is not waterfall theatre. It is a living, versioned, reviewable artifact that answers:

- what problem we are solving;
- who it matters to;
- what is in and out of scope;
- which behaviors must exist;
- which technical, legal, or product constraints must be respected;
- which criteria prove the work is correct;
- which risks require human review.

The central shift is simple: the agent does not implement "the idea that was implicit in the chat." It implements against a contract.

## Anatomy of a useful spec

A useful agent-facing spec usually has seven blocks.

### 1. Problem and expected outcome

Explain the business or user outcome, not only the technical task.

Weak: "add auth".

Better: "enterprise users need to sign in through the identity provider configured by their organization, without breaking email login for existing accounts".

### 2. Scope and non-goals

Say what is included and what is excluded. This prevents the agent from "helping too much".

Example: "implement callback and error screen; do not migrate existing users; do not change billing plans".

### 3. Interfaces and contracts

List relevant endpoints, events, schemas, components, permissions, states, and integrations. The clearer the boundaries, the less the agent invents architecture.

### 4. Acceptance criteria

Acceptance criteria are tests in human language. They should be verifiable.

Example: "when the provider returns a domain-not-allowed error, the user sees a recoverable message and no new user is created".

### 5. Constraints

Include boundaries such as "do not add a dependency", "do not change the public API", "keep Safari compatibility", or "do not persist tokens in localStorage".

### 6. Validation plan

Define which commands, manual checks, QA scenarios, security checks, and evidence must exist before the PR is accepted.

### 7. Risks and rollback

For enterprise work, the question is not only "does it work?". It is "if it fails, how do we notice and how do we roll back?".

## What SDD is not

SDD is not writing three pages to change button copy. It is also not freezing requirements as if nothing will change.

The spec should be proportional to risk. A simple visual tweak may need a five-line spec. A change to authentication, payments, personal data, or infrastructure deserves more care.

SDD does not replace judgment either. A bad spec simply produces a bad implementation with more confidence theater. The value is making intent explicit so humans and agents can review the same thing.

## What Harness Design is

Harness Design is the design of the **agent operating environment**.

If SDD answers "what must be true?", the harness answers "how do we create a system where the agent can execute, validate, and improve without relying on informal memory?".

A harness can include:

- persistent instructions such as `AGENTS.md`, `CLAUDE.md`, or `.cursor/rules/`;
- versioned architecture, product, and domain documentation;
- spec, PRD, plan, and review templates;
- tools available to the agent, such as terminal, GitHub, browser, logs, and MCP servers;
- permission boundaries and human approval points;
- validation commands, CI, lint, typecheck, tests, and security checks;
- PR standards, ownership, rollback, and audit practices;
- learning loops that turn failures into rules, tests, or docs.

The goal is not to trap the agent. It is to create rails. Rails do not prevent movement; they make movement predictable.

## How SDD and Harness Design work together

Think of it this way:

| Layer      | SDD defines                             | Harness ensures                                  |
| ---------- | --------------------------------------- | ------------------------------------------------ |
| Intent     | problem, scope, and expected behavior   | templates and rituals to capture it consistently |
| Context    | which files, APIs, and decisions matter | docs, rules, and repo maps the agent can access  |
| Execution  | which changes should happen             | tools, permissions, and task plans               |
| Validation | how we know it is correct               | tests, lint, typecheck, CI, and evidence         |
| Governance | risk, approval, and rollback            | PRs, ownership, audit trails, and policies       |
| Learning   | what the spec left ambiguous            | updates to rules, docs, and checks               |

SDD without a harness becomes pretty documentation. A harness without SDD becomes automation without direction. A professional flow needs both.

## Applied example: enterprise SSO

Imagine a company needs to add SSO login for enterprise customers.

### Amateur flow

Someone asks the agent: "implement SSO with Google". The agent creates an integration, adds dependencies, changes auth, passes the build, and the screen opens. It looks like progress.

But nobody answered:

- which providers are supported;
- how domains map to organizations;
- what happens to existing users;
- who can configure SSO;
- how login is audited;
- how to roll back if the provider fails;
- which tests prove permissions do not leak.

That is not enterprise engineering. It is a prototype in a blazer.

### SDD + Harness flow

**PM** writes the intent: enterprise customers need to authenticate users through the SSO provider configured for their organization.

**Tech lead** turns that into a spec: flows, states, permissions, boundaries, schemas, endpoints, and risks.

**QA** derives scenarios: user without an allowed domain, provider unavailable, expired invitation, organization without SSO, existing user with the same email.

**Security** defines constraints: tokens cannot go into unsafe storage; logs cannot contain secrets; callback must validate state.

**Agent** receives the spec, auth-module context, project rules, and validation commands.

**Harness** runs the gates: typecheck, lint, unit tests, integration tests, build, diff review, and security checklist.

**PR** includes evidence: what changed, which commands ran, which risks remain, and how to roll back.

If something fails, the team does not just ask the agent to "try again". It asks: was an acceptance criterion missing? A test? A rule? A tool? The answer becomes an improvement to the harness.

## Where this breaks

- **Spec theater**: writing a vague spec just to look mature. If it does not guide implementation or validation, it is not useful.
- **Premature harnessing**: creating rules, hooks, and automations before understanding the manual flow. Automating a bad process only scales noise.
- **Too much context**: dumping all documentation into the agent. A good harness works through maps and links, not infinite encyclopedias.
- **Permission without boundaries**: giving broad access to sensitive tools without policy, logs, or risk-appropriate review.
- **Surface-level validation**: running build and calling it done. Build proves compilation, not product correctness.
- **Spec drift**: code changes, the spec does not. After a few deliveries, the contract becomes fiction.
- **Confusing autonomy with abandonment**: good autonomy is delegation with evidence. Abandonment is accepting output without responsibility.

## How to start without bureaucracy

You do not need an agent laboratory before using this.

Start small:

1. Create a simple spec template for medium-risk features.
2. Write a short `AGENTS.md` with commands, architecture, and essential rules.
3. Define the minimum bar before accepting a PR: diff read, lint, typecheck, tests, and acceptance criteria.
4. For each recurring agent failure, choose an action: improve the spec, add a test, update a rule, or create a script.
5. Review the harness once per sprint or before major delivery.

The goal is not bureaucracy. The goal is reducing improvisation where improvisation is expensive.

## What to take with you

- SDD turns intent into a verifiable contract.
- Harness Design turns that contract into execution, validation, and governance.
- In enterprise teams, quality comes from the system, not from one brilliant prompt.
- The higher the risk, the more explicit the spec, context, validation, and ownership need to be.
- The best harness is alive: it learns from failures, removes useless rules, and makes work more legible for humans and agents.
